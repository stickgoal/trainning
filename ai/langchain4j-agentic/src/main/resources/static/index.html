🧪 六种工作流模式

点击下方卡片中的「发送请求」按钮，实时调用后端 API 查看效果

①
Sequential — 顺序审批链
RiskAgent → FinanceAgent → ServiceAgent

用户申请退款，依次经过风控检查、财务核算、客服确认三级审批，每步依赖前一步的输出结果。

flowchart LR A[退款申请] --> B[RiskAgent
风控检查] B --> C[FinanceAgent
财务核算] C --> D[ServiceAgent
客服确认] D --> E[最终结果]
POST /api/sequential/refund
②
Parallel — 并行核验
OrderCheck + CreditCheck + StockCheck → Summary

退款申请同时核验订单状态、用户信用、商品库存三个维度，互不依赖，并行处理后由 SummaryAgent 汇总决策。

flowchart TD A[退款申请] --> B1[OrderCheck] A --> B2[CreditCheck] A --> B3[StockCheck] B1 & B2 & B3 --> C[SummaryAgent] C --> D[最终决策]
POST /api/parallel/check
③
Loop — 迭代优化
DraftAgent ↔ ReviewAgent 循环直到评分达标

自动生成售后回复文案，通过"起草→评审→修改"循环迭代，直到 ReviewAgent 评分 ≥ 8 分后输出最终文案。

flowchart TD A[投诉] --> B[DraftAgent
起草] B --> C[ReviewAgent
评审] C -->|≥8分| D[输出文案] C -->|需改进| B
POST /api/loop/reply
④
Conditional — 条件分流
ClassifyAgent → 自动退款 / 人工审核 / 换货处理

根据退款原因自动分类，走不同处理通道：质量问题→快速退款、个人原因→人工审核、换货请求→换货处理。

flowchart TD A[退款申请] --> B[ClassifyAgent] B --> C{分类结果} C -->|质量问题| D[AutoRefund] C -->|个人原因| E[ManualReview] C -->|换货| F[Exchange]
POST /api/conditional/refund
⑤
Supervisor — 主管调度
LLM 驱动动态编排 Emotion/Fact/Solution/Notify Agent

复杂售后诉求由 Supervisor Agent 动态编排执行计划，自主决定调用顺序，灵活调度 Emotions、Fact、Solution、Notify 四个子 Agent。

flowchart TD A[复杂投诉] --> B[Supervisor] B -.-> C[EmotionAgent] B -.-> D[FactAgent] B -.-> E[SolutionAgent] B -.-> F[NotifyAgent] C & D & E & F -.-> B B --> G[通知文案]
POST /api/supervisor/handle
⑥
HumanInTheLoop — 人工介入
PreCheck → 暂停等待审批 → Execute

大额退款需人工审批。Agent 完成前置检查后自动暂停，等待主管审批后恢复执行。两步交互：提交退款 → 人工审批。

flowchart TD A[大额退款] --> B[PreCheckAgent] B --> C[⏸️ 暂停等待] C -->|APPROVED| D[ExecuteAgent] C -->|REJECTED| E[拒绝退款] D --> F[最终结果]
⚠️ 两步操作：先点 提交退款 获取 requestId，再点 人工审批 完成流程
POST .../refund
POST .../approve
